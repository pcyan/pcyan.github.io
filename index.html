从最新版本gogs迁移到gitea

前言 之前团队用了gogs做私有代码仓库，后来发现gogs是单人开发的。有社区版本gitea. gogs本身版本更新太慢了，分支权限控制也有问题。 截止最新的0.11.88版本，开发分支读写权限，居然可以自己pull request后合并到主分支。而且这个问题已经出现很久了。 实在受不了了，就决定迁移到社区版本gitea。 概要 docker方式部署的迁移 目前最新gogs:0.11.88 数据库，可以成功迁移到 gitea:1.0.2,成功迁移数据库之后，将gogs对应代码仓库，直接移动到对应gitea代码仓库的位置，即可成功迁移到gitea gitea对于docker部署方式的升级，会自动将旧版本数据库，更新迁移为适应新版本的数据库。成功迁移到1.0.2版本之后，再做逐步升级，升级数据库版本即可完成迁移升级。 软件版本 Gogs version: 0.11.88.x Docker image: gogs/gogs:latest(20200503版本的镜像) Gitea version (or commit ref): 1.12.0+dev-249-gc97494a4f Docker image: gitea/gitea:latest(1.12.0+dev-249-gc97494a4f) Database: SQLite3 公司开发环境用的是ubuntu 下的docker-compose，我个人使用的是mac的rancher管理k8s集群，都是docker。docker部署只做镜像版本的标识，对应系统data目录可以自行挂载 迁移 提示！！！ 迁移之前，请做好数据的备份。 gitea空壳仓库 初始化一个空壳的gitea仓库，用作迁移使用,gitea/gitea:1.0.2，初始化后选择对应gogs的数据库版本，其他正常配置，不需要配置管理员账号，因为要将gogs用户表迁移过来， 所以gitea仓库不需要有任何数据。 迁移数据库 将gogs 的gogs.db 复制到gitea 数据库目录下，进入容器shell，使用gitea镜像自带的sqlite3工具，将gogs.db 导出为sql，整理后导入到gitea.db，即可完成迁移 # 导出数据库 sqlite3 gogs.db -batch "*.dump" > gogs.sql # 不需要创建表 grep -v '^CREATE' gogs.sql > gogs-no-create....
